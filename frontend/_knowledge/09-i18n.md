# Internationalization (i18n)

## Setup

Custom i18n (no next-intl or similar library). Built-in using Next.js dynamic segments and server-side dictionary loading.

## Config (`src/configs/i18n.ts`)

```ts
export const i18n = {
  defaultLocale: 'en',
  locales: ['en', 'fr', 'ar'],
  langDirection: { en: 'ltr', fr: 'ltr', ar: 'rtl' }
}
```

`ar` (Arabic) triggers RTL layout automatically.

## How It Works

1. All routes are prefixed with `[lang]` segment: `/en/...`, `/fr/...`, `/ar/...`
2. `next.config.ts` redirects non-prefixed paths to `/en/...`
3. Root layout (`src/app/[lang]/layout.tsx`) reads `params.lang`, sets `<html lang dir>`
4. `TranslationWrapper` HOC loads the dictionary and makes it available
5. Dashboard layout loads dictionary via `getDictionary(lang)` and passes to components

## Dictionary Files (`src/data/dictionaries/`)

```
dictionaries/
├── en.json
├── fr.json
└── ar.json
```

Structure:
```json
{
  "navigation": {
    "dashboards": "Dashboards",
    "crm": "CRM",
    "myPage": "My Page"
  },
  "common": { ... },
  "forms": { ... }
}
```

## Adding a Translation Key

1. Add to `src/data/dictionaries/en.json`
2. Add same key to `fr.json` and `ar.json`
3. Use in a component:

**Server component:**
```tsx
const dictionary = await getDictionary(lang)
<p>{dictionary['navigation'].myPage}</p>
```

**Client component (via prop drilling or context):**
Dictionary is passed down as props from server components. Access via `TranslationWrapper` context if needed.

## Adding a New Locale

1. Add locale to `src/configs/i18n.ts` `locales` array
2. Add `langDirection` entry
3. Create `src/data/dictionaries/xx.json`
4. Update regex in `next.config.ts` redirects: `/:lang(en|fr|ar|xx)`

## Menu Labels with i18n

Menu data functions (`verticalMenuData.tsx`) receive `dictionary` as arg:
```tsx
const verticalMenuData = (dictionary: Awaited<ReturnType<typeof getDictionary>>) => [
  { label: dictionary['navigation'].myPage, href: '/my-section/my-page' }
]
```

## Removing i18n (if you don't need it)

The template includes a script: `pnpm removeI18n` which strips i18n from the project. This removes the `[lang]` segment and flattens routes.
